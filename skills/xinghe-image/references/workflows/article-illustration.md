# 文章配图子流程

> **核心理念**：根据章节内容和配图位置，灵活选择「表达性配图」或「说明性配图」。
>
> - **表达性配图**（默认优先）：海报级视觉作品，用视觉隐喻/情绪传达内容，作为章节分隔，无文字标签
> - **说明性配图**：信息图形式，含关键文字标注、流程箭头、数据可视化，适合结构性强的内容
>
> **执行框架**：复用 baoyu-article-illustrator 的文章分析 + outline + 提示词文件管理流程，
> 在视觉概念提炼阶段判断每张图适合哪种类型。

---

## 进度 Checklist

```
文章配图进度：
- [ ] Step 1: 文章深度分析（结构 + 位置识别 + 视觉概念提炼）
- [ ] Step 2: 风格确认 ⚠️ BLOCKING（来自 SKILL.md Step 2）
- [ ] Step 3: 生成 outline.md ⚠️ 用户确认后才能生图
- [ ] Step 4: 生成所有提示词文件 ⛔ BLOCKING（先于生图）
- [ ] Step 5: 批量生图
- [ ] Step 6: 回写文章（插入图片引用）
```

## Step 1：文章深度分析

### 1.1 基础信息提取

```
- 文章标题
- 估算字数
- 文章类型（技术/观点/故事/教程/分析报告/叙事）
- 目标读者
- 整体情感基调（正向激励/沉思批判/紧迫警示/轻松幽默/中性呈现）
- 文章节奏（平缓/递进/跌宕）
```

### 1.2 章节结构解析

将文章分解为层级结构：

```
一级：文章整体论点/主题
  └─ 二级：主要章节（按标题或自然段落划分，3-7个）
       └─ 三级：每个章节的核心信息点
            └─ 四级：关键概念/数字/转折点
```

**章节类型识别**（影响配图情绪选择）：

| 章节类型 | 配图情绪建议 |
|---------|------------|
| 引言/问题提出 | 神秘感、悬念感、引发好奇 |
| 现状描述/数据呈现 | 中性观察、静止感 |
| 核心论点/转折 | 强烈冲击、情绪爆发点 |
| 案例/故事 | 叙事感、情境感、氛围感 |
| 解决方案/方法论 | 清晰结构感、秩序感 |
| 总结/展望/行动号召 | 升华感、激励感、开放性 |

### 1.3 插图位置决策

**该放图的位置**：

- 文章核心论点所在段落（必须）
- 抽象概念需要视觉锚点的地方
- 叙事节奏转折点（情绪需要「停顿感」）
- 长段落中读者注意力开始衰减的地方

**不放图的位置**：

- 仅是过渡/连接性段落
- 已经有具体数据图/截图的地方
- 文字本身已足够有画面感

**数量参考**：

| 文章字数 | 推荐配图数 | 说明 |
|---------|----------|------|
| < 1500 字 | 1-2 张 | 开头（引入情绪）+ 结尾（升华） |
| 1500-4000 字 | 3-4 张 | 开头 + 核心论点 + 结尾 |
| 4000-8000 字 | 4-6 张 | 主要章节各一张 |
| > 8000 字 | 6-8 张 | 每个主要章节 + 关键转折点 |

### 1.4 视觉概念提炼（关键步骤）

对每个配图位置，同时判断**配图类型**和**视觉概念**。

#### 配图类型判断

| 章节特征 | 推荐类型 | 说明 |
|---------|---------|------|
| 引言/问题/情绪转折 | **表达性** | 用视觉隐喻引发共鸣，无需文字 |
| 数据/统计/趋势对比 | **说明性** | 数字和图表是内容本身，需要标注 |
| 流程/步骤/方法论 | **说明性** | 流程箭头和步骤标签是必要信息 |
| 抽象观点/核心论点 | **表达性** | 隐喻比图表更有冲击力 |
| 技术架构/系统关系 | **说明性** | 结构图需要节点标签才能理解 |
| 叙事/故事/氛围 | **表达性** | 氛围感优先，文字会破坏沉浸感 |
| 对比/优缺点分析 | **二选一** | 数据型对比用说明性；感性对比用表达性 |

#### 表达性概念提炼原则

- 不是章节内容的文字总结
- 而是可以用单一图像独立表达的**情绪主题**或**视觉隐喻**
- 图像完全自洽，不需要文字说明才能理解

**表达性 vs 说明性示例**：

| 文章内容 | 说明性配图 | 表达性配图 | 推荐 |
|---------|----------|----------|------|
| 算法推荐侵蚀注意力 | 推荐算法流程图 | 被碎片屏幕切割的人脸 | 表达性 |
| AI 技术加速发展 | AI 发展时间轴 | 光流穿越黑暗隧道的第一视角 | 表达性 |
| 三步操作方法论 | 步骤流程图（含标注） | 抽象的三段式构图 | **说明性** |
| 数据增长 42% | 数字大字排版信息图 | 膨胀爆破的几何体 | **说明性** |
| 数据隐私泄露风险 | 数据泄露示意图 | 溶解的数字面具 | 表达性 |
| 团队协作的重要性 | 团队结构组织图（有节点标签） | 相互咬合旋转的齿轮群 | 二选一 |

## Step 2：风格确认（来自 SKILL.md Step 2）⚠️ BLOCKING

风格由 SKILL.md Step 2 的组装路线流程完成（从 `styles-index.csv` 推荐 3 个方向，用户确认），此处接收确认结果后进入 Step 3。

**接收参数**：

- `{style}` = 用户确认的风格，对应 `styles-index.csv` 中该风格行的 `style_en` 字段值（英文名），例如 `"Risograph"`、`"Neubrutalism"`
- STYLE_SECTION、PALETTE_SECTION 所需字段（`rendering_keywords`、`visual_rule`、`color_rule`、`title_style`）在 Step 4 生成提示词时，通过 `{style}` 从 `styles-index.csv` 对应行读取

**若 SKILL.md Step 2 还未执行**（用户直接进入此子流程时），则在此处执行组装路线的风格推荐步骤，再继续 Step 3。

## Step 3：生成 outline.md ⚠️ 用户确认后才能生图

基于 Step 1 + Step 2 的结果，生成配图计划，**展示给用户确认**，确认后才能进入生图。

保存到 `{output-dir}/outline.md`：

```yaml
---
style: "[确认的风格名]"
image_count: [N]
article: "[文章标题或文件名]"
---

## 配图 1

**位置**: [章节标题] / [段落首句或位置描述]
**插入理由**: [为什么这里需要一张图]
**配图类型**: [表达性 / 说明性]
**视觉概念**: [表达性：视觉隐喻/情绪主题 | 说明性：信息结构描述]
**情绪基调**: [神秘/冲击/叙事/升华/结构/中性...]
**文件名**: 01-[slug].png

## 配图 2

**位置**: [...]
**插入理由**: [...]
**配图类型**: [...]
**视觉概念**: [...]
**情绪基调**: [...]
**文件名**: 02-[slug].png

...
```

**展示给用户**：

```
📋 配图计划（共 [N] 张）— 风格：[风格名]

[直接粘贴 outline.md 内容]

确认开始生图？或告诉我需要调整的地方。
```

## Step 4：生成提示词文件 ⛔ BLOCKING

**所有提示词文件必须先保存，才能开始生图。**

为每张配图创建 `{output-dir}/prompts/NN-[slug].md`，根据配图类型使用不同模板：

### 表达性配图提示词模板

```markdown
---
illustration_id: 01
type: expressive
style: "[风格名]"
size: "1264x848"
---

[风格描述，30-50字，取自 styles-index.csv 的 rendering_keywords + visual_rule] 风格的横版海报作品。
[情绪/氛围描述，20-30字]。
[单一视觉主体描述，30-50字，单一焦点，强视觉冲击]。
强视觉冲击，单一焦点构图，海报构图感，无文字标签，无流程箭头，无数据图表。
[色彩要求，参考所选风格的色彩规范，含具体色号]。
```

### 说明性配图提示词模板

```markdown
---
illustration_id: 02
type: informational
style: "[风格名]"
size: "1264x848"
---

[风格描述，30-50字] 风格的横版信息图海报。
[整体信息主题，一句话]。
[具体信息内容：核心信息点、数字、关系结构，30-60字]。
清晰的信息层级，横版宽幅构图，图形化表达，视觉流向从左到右。
[如有文字标签需求：包含简短中文关键词标注，字体清晰可读]。
[色彩规范，含具体色号]。
```

**提示词通用要求**：

- **全程中文**（星河模型中文效果更好）
- **色彩具体化**：用色号（如 `#00B4FF`）+ 中文名（「霓虹蓝」）组合描述
- 表达性：**单一视觉主体**，画面只有一个绝对主体，不堆砌元素
- 说明性：**信息层级清晰**，文字标签简短（不超过5字/条）

**提示词示例 — 表达性**（Risograph 孔版印刷风 + 「注意力被碎片化」概念）：

```
孔版印刷风（Risograph）风格的横版海报作品，套印偏差效果，颗粒感网点纹理，颜色叠加产生第三色，限色印刷感。
压迫感与碎裂感并存的情绪，注意力消解的窒息感。
一张人脸被无数半透明的矩形碎片分割打散，碎片边缘有套印偏差的颗粒质感，中心的眼睛仍然清晰，其余区域正在消解。
强视觉冲击，单一焦点构图，海报构图感，无文字标签，无流程箭头，无数据图表。
荧光橙 #FF6B35 + 荧光蓝 #0099CC 双色叠加，叠加区域产生深棕色，米色纸底 #F5F0E8。
```

**提示词示例 — 说明性**（Neubrutalism 风格 + 「三步操作流程」）：

```
新暴力主义（Neubrutalism）风格的横版信息图海报，粗黑边框，高对比度色块，原始字体感。
展示产品接入的三步操作流程。
横向三个步骤节点：「1. 申请 API Key」→「2. 配置环境变量」→「3. 调用接口」，箭头连接，每步配简短说明文字。
清晰的信息层级，横版宽幅构图，视觉流向从左到右，步骤节点用色块区分。
包含简短中文关键词标注，字体清晰可读。
主色调：纯黑 #000000 + 电光黄 #FFE500 + 纯白 #FFFFFF，重色块分区。
```

## Step 5：批量生图

提示词文件全部就绪后，依次执行：

```bash
# {baseDir} = SKILL.md 所在目录

python3 {baseDir}/scripts/generate.py \
  --prompt "$(cat {output-dir}/prompts/01-[slug].md | grep -v '^---' | grep -v '^illustration_id' | grep -v '^style' | grep -v '^size')" \
  --size 1264x848 \
  --output {output-dir}/01-[slug].png
```

更简洁的方式——直接读提示词正文（frontmatter 以下部分）：

```bash
# 每张图执行一次，报告进度
python3 {baseDir}/scripts/generate.py \
  --prompt "[从提示词文件读取的中文正文]" \
  --size 1264x848 \
  --output {output-dir}/NN-[slug].png
```

每张生成后报告：`✓ 配图 [N/总数]: [slug] → [路径]`

失败时：重试一次，仍失败则记录并继续后续图片。

## Step 6：回写文章

生图完成后，在文章对应位置插入图片引用。

**插入格式**：

```markdown
![{视觉概念简短描述}]({相对路径}/NN-[slug].png)
```

**路径规则**：相对于文章文件的相对路径。

**示例**：

```markdown
...上一段文字...

![注意力碎片化的视觉隐喻](imgs/01-attention-fragments.png)

...下一段文字...
```

## 文件目录结构

```
{output-dir}/                    # 默认为文章同目录下的 imgs/ 子目录
├── outline.md                   # 配图计划（Step 3 生成，用户确认）
├── prompts/
│   ├── 01-[slug].md             # 提示词文件（Step 4，先于生图）
│   ├── 02-[slug].md
│   └── ...
├── 01-[slug].png                # 生成的图片
├── 02-[slug].png
└── ...
```

**Slug 规则**：提取视觉概念 2-4 词，kebab-case。
示例：「被碎片切割的人脸」→ `face-fragments`

**备份规则**：文件已存在 → 自动备份为 `*-backup-YYYYMMDD-HHMMSS`

## 中文适配注意事项

1. **提示词全程中文**，包括风格描述、色彩描述、构图要求
2. **色彩描述**：中文名 + 色号组合，如「霓虹蓝 #00B4FF」
3. **表达性配图**：提示词中明确排除文字元素，保持纯视觉叙事
4. **说明性配图中的文字**：要求「简短中文关键词标注」，避免要求复杂段落——模型生成的汉字可能有错误，标注尽量控制在 5 字以内
5. **`revised_prompt`**：PE 模型会返回优化后的提示词，可参考用于下次迭代，但不需要用来覆盖当前生成
