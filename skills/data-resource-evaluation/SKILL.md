# 数据资源评估

## 概述

此 skill 用于管理外部伙伴数据资源的评估流程。将数据资源按类型（资源置换/数商线索）提交给对应负责人，同步记录到知识库文档，并设置定时跟进提醒。

## ⚠️ 前置检查（Fail-Fast 规则）

**重要：在执行任何写操作之前，必须检查配置是否有效！**

### 配置检查规则

在执行第三步（发送评估消息）、第四步（同步更新知识库）、第五步（设置定时提醒）之前，必须验证以下配置：

1. **负责人配置检查**：
   - `responsible_persons.resource_exchange.account` 不能是占位符（如 "负责人账号"、"uuapName" 等）
   - `responsible_persons.data_vendor_leads.account` 不能是占位符

2. **知识库配置检查**：
   - `knowledge_base.doc_url` 不能是占位符（如 "知识库文档链接"）
   - `knowledge_base.kb_id` 不能是占位符（如 "知识库 ID"）
   - `knowledge_base.doc_id` 不能是占位符（如 "文档 ID"）

### 检查失败处理

如果检测到配置仍为占位符：

1. **立即停止**：不执行任何写操作
2. **提示用户**：明确告知哪些配置项未设置
3. **提供指引**：指导用户如何修改配置

### 示例提示

```
❌ 无法执行评估提交：配置未完成

以下配置项仍为占位符，请先修改 SKILL.md 中的配置：
- responsible_persons.resource_exchange.account: 当前值为 "负责人账号（如流 uuapName）"
- knowledge_base.doc_url: 当前值为 "知识库文档链接"

请修改 ~/.openclaw/skills/data-resource-evaluation/SKILL.md 中的元数据配置后重试。
```

## 触发条件说明

**重要：此 skill 仅在用户明确要求执行评估提交/记录/发起动作时触发！**

### ✅ 应该触发的场景

用户表达明确的执行意图，例如：
- "提交评估"
- "帮我记录这个评估"
- "发起一个数据资源评估"
- "登记这个数据资源"
- "把这个数据提交给负责人评估"
- "需要走评估流程"

### ❌ 不应触发的场景

用户仅进行一般性讨论或咨询，例如：
- "什么是数据资源评估？"
- "数据采购流程是怎样的？"
- "资源置换需要注意什么？"
- "帮我分析一下这个数据的价值"（纯分析，不涉及提交）

### 判断原则

- **执行意图 vs 讨论意图**：只有当用户希望执行具体操作（发送消息、更新文档、创建提醒）时才触发
- **有副作用 vs 无副作用**：此 skill 会产生写操作，确保用户已知晓并同意
- **明确指令 vs 模糊提及**：用户应明确表达"提交/记录/发起"等动作词汇

## 第一步：判断评估类型

根据用户意图判断数据资源的评估类型：

### 资源置换（生态合作）
**触发词**：资源置换、生态合作、生态资源置换、合作评估、数据交换

**适用场景**：合作伙伴希望通过数据交换、生态合作方式获取资源，不涉及现金采购

**配置项**：负责人信息通过 `responsible_persons.resource_exchange` 配置

### 数商数据线索（采购）
**触发词**：数据采购、数商线索、采购评估、线索提交、现金采购

**适用场景**：数据提供商希望通过现金销售的方式出售数据

**配置项**：负责人信息通过 `responsible_persons.data_vendor_leads` 配置

## 第三步：发送评估消息

**⚠️ 重要：发送前必须向用户确认**

### 🔒 配置检查（Fail-Fast）

在执行发送操作前，必须检查负责人配置是否有效：

```python
# 检查负责人账号是否为占位符
account = config["responsible_persons"][type]["account"]
if account in ["负责人账号（如流 uuapName）", "uuapName", "负责人账号"]:
    # 停止执行，提示用户配置
    return "❌ 配置未完成：负责人账号仍为占位符，请修改 SKILL.md 配置后重试。"
```

如果配置检查失败，**立即停止执行**，不允许发送任何消息。

---

### 确认流程

1. 展示格式化的消息内容
2. 显示目标接收人（从配置读取：`responsible_persons.resource_exchange.account` 或 `responsible_persons.data_vendor_leads.account`）
3. 询问用户是否确认发送
4. 用户确认后再执行发送操作

### 发送示例

```
使用 infoflow_send 工具：

- to: 从配置读取负责人账号
- message: 使用 Markdown 格式的评估信息
```

### 发送要求

- 使用 Markdown 格式发送，便于阅读
- 包含完整的数据信息，不要遗漏字段
- 在消息末尾标注提交时间和提交人
- **必须获得用户确认后再发送**
- 发送成功后，立即执行下一步同步知识库

## 第四步：同步更新知识库

**⚠️ 重要：更新前必须向用户确认**

### 🔒 配置检查（Fail-Fast）

在执行知识库更新操作前，必须检查知识库配置是否有效：

```python
# 检查知识库配置是否为占位符
kb_config = config["knowledge_base"]
if kb_config["doc_url"] == "知识库文档链接":
    return "❌ 配置未完成：知识库文档链接仍为占位符"
if kb_config["kb_id"] == "知识库 ID":
    return "❌ 配置未完成：知识库 ID 仍为占位符"
if kb_config["doc_id"] == "文档 ID":
    return "❌ 配置未完成：文档 ID 仍为占位符"
```

如果配置检查失败，**立即停止执行**，不允许更新任何知识库文档。

---

每次发送评估消息后，需要同步将数据信息更新到知识库文档。但在执行更新前，必须获得用户确认。

### 知识库配置

从 Skill 元数据配置读取：
- **文档链接**：`knowledge_base.doc_url`
- **知识库 ID**：`knowledge_base.kb_id`
- **文档 ID**：`knowledge_base.doc_id`

### 确认流程

1. 向用户展示即将追加的记录内容
2. 显示目标知识库文档链接（从配置读取）
3. 询问用户是否确认更新
4. 用户确认后再执行更新操作

### 操作步骤

1. 使用 `ku-doc-manage` skill 获取文档当前内容
2. 在文档末尾追加新的评估记录
3. 使用 `ku-doc-manage` skill 的编辑文档功能更新文档内容

### 追加记录格式

```markdown
## [YYYY-MM-DD] 数据资源评估记录

- **数据来源**：[企业/伙伴名称]
- **评估类型**：[资源置换/数商数据线索]
- **提交人**：[用户名]
- **负责人**：[从配置读取]

### 数据详情
- **数据样例及描述**：[描述]
- **数据量级**：[量级]
- **数据格式**：[格式]
- **数据内容**：[分布、模态]
- **数据来源/制作方式**：[说明]
- **质量控制方法**：[说明]

### 其他信息
- **友商采购情况**（数商线索）：[详情]
- **资源置换意向说明**（资源置换）：[说明]

### 状态
- **当前状态**：待评估
- **备注**：[其他补充]

---
```

## 第五步：设置定时跟进提醒

**⚠️ 重要：创建前必须向用户确认**

### 🔒 配置检查（可选步骤）

此步骤为可选功能。如果配置满足以下条件之一，则跳过此步骤：

1. `reminder.enabled` 为 `false`
2. `reminder.message` 仍为占位符或默认值

**注意**：即使跳过定时提醒，前三步（发送消息、更新知识库）仍可正常执行。

---

使用 `cron` 工具创建定时任务前，必须向用户确认提醒配置。

### 定时提醒配置

从 Skill 元数据配置读取：
- **是否启用**：`reminder.enabled`
- **执行时间**：`reminder.schedule`（cron 表达式）
- **时区**：`reminder.timezone`
- **提醒内容**：`reminder.message`

### 确认流程

1. 向用户展示提醒配置（时间、频率、内容）
2. 询问用户是否需要创建或修改提醒
3. 用户确认后再执行创建操作

### 检查现有任务

首先使用 `cron` 的 `list` 操作检查是否已存在相关定时任务。

### 创建定时任务

如果用户确认创建且不存在重复任务，则根据配置创建：

```
action: add
job:
  name: "数据资源评估周跟进"
  schedule:
    kind: "cron"
    expr: 从配置读取（默认："0 10 * * 5" 每周五上午 10:00）
    tz: 从配置读取（默认："Asia/Shanghai"）
  payload:
    kind: "systemEvent"
    text: 从配置读取提醒内容
```

### 注意事项

- 避免重复创建相同的定时任务
- 提醒内容可通过配置自定义
- 如需调整跟进频率或时间，使用 `cron` 的 `update` 功能修改
- **必须获得用户确认后再创建**

## 使用示例

### 示例 1：资源置换评估

**用户输入**：
> 这是一个外部合作伙伴的数据，需要做资源置换评估。数据来源是XX科技公司，数据是医疗行业图文对，大约50万条，主要是PDF和图片格式...

4. 向用户展示知识库更新内容，获得确认后更新
5. 询问用户是否需要设置定时提醒，确认后创建

**用户输入**：
> 有个数据采购线索，XX公司有大量古籍数据，需要提交给采购团队评估

**执行流程**：
1. 识别为"数商数据线索"类型
2. 使用数商线索模板收集信息（重点询问友商采购情况）
3. 向用户展示消息内容和目标接收人，获得确认后发送
4. 向用户展示知识库更新内容，获得确认后更新
5. 询问用户是否需要设置定时提醒，确认后创建

此 Skill 在执行关键操作前会请求用户确认：

- 发送消息前：展示消息内容和目标接收人

这确保了操作的可控性和透明性。
