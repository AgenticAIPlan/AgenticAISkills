# 错误处理说明

本文档说明脚本可能遇到的失败类型，以及 Agent 应如何向用户反馈。

## 失败类型

| 类型 | 常见原因 | 脚本行为 | 用户反馈要求 |
|------|----------|----------|--------------|
| 输入错误 | PDF 不存在、路径不是文件、扩展名不是 `.pdf`、目录中没有 PDF | 返回 `ok=false` 的 JSON 摘要，退出码为 1 | 指出具体路径和修正方式 |
| 输出冲突 | 同名 `paper.md` 或 `paper/` 已存在 | 默认失败，不覆盖 | 询问是否允许 `--force` |
| 配置错误 | 未配置 `PADDLEOCR_DOC_PARSING_API_URL` 或 `PADDLEOCR_ACCESS_TOKEN` | 返回 PaddleOCR 包装错误 | 要求用户在运行环境配置变量，不要把密钥提交进仓库 |
| 接口错误 | 鉴权失败、限流、服务端错误、接口超时 | 返回接口错误码或错误消息 | 保留原始错误，不切换到其他 OCR 流程 |
| 返回结构异常 | PaddleOCR 响应缺少 `layoutParsingResults` 或 `markdown.text` | 返回结构校验错误 | 说明可能是接口版本或输入文件质量问题 |
| 图片下载失败 | 图片 URL 超时、403、404 或网络错误 | 当前 PDF 失败；批量模式按策略停止或继续 | 报告失败 PDF，并说明 Markdown 未发布为成功结果 |

## 批量模式错误策略

默认不加 `--continue-on-error`，脚本遇到第一个失败项后停止，最终摘要中：

- `ok` 为 `false`
- `failed` 至少为 1
- `errors` 包含失败 PDF 和错误原因

加上 `--continue-on-error` 后，后续 PDF 会继续处理。Agent 最终必须同时说明：

- 成功处理了多少个 PDF
- 失败了多少个 PDF
- 每个失败 PDF 的原因

## 禁止行为

- 不要在失败后手工编辑 Markdown 假装成功。
- 不要把失败的 PDF 从统计中删除。
- 不要把 token、`.env` 或接口地址写入 Skill 内容、PR 描述或提交记录。
- 不要自动启用 `--force` 覆盖已有输出。
- 不要改用其他 OCR、截图解析或手工复制文本作为 fallback。

## 最小反馈格式

当任务失败或部分失败时，向用户反馈至少包含：

```text
总数：N
成功：N
失败：N
失败明细：
- /path/to/paper.pdf：错误原因
```
