# 项目评审快速检查清单

## 第一步：快速筛选（30秒）

### 必查项（任一不合格则直接反馈，无需评分）
- [ ] 项目是否为公开状态
- [ ] 是否使用飞桨或文心大模型技术
- [ ] README是否有实质内容（至少50字，包含基本说明）
- [ ] 是否有代码文件（至少3个文件或100行有效代码）
- [ ] 是否有明显的水文特征（完全复制、广告、灌水等）

### 水文判定特征（≥3个即为水文）
- [ ] README内容少于50字
- [ ] 只有README没有代码
- [ ] 代码文件少于3个且总代码行数少于100行
- [ ] 项目标题与内容完全不相关
- [ ] 完全复制粘贴官方示例无任何修改
- [ ] 明显的广告或引流内容

**如果以上任一项不合格，直接反馈问题，跳过后续流程。**

---

## 第二步：AI生成内容快速检测（1分钟）

### 高疑似AI生成特征（任一即标记）
- [ ] README完全通用模板化，缺乏具体项目细节
- [ ] 代码注释高度统一、格式过于完美
- [ ] 代码结构和变量命名过于规整，缺少实际业务场景
- [ ] 文档中充斥"全面"、"强大"、"高效"等大词但缺乏数据支撑
- [ ] 多处出现"在本文中"、"值得注意的是"、"总之"等AI常见表达

### 中等疑似AI生成特征（≥3个标记为疑似）
- [ ] README结构过于标准（Introduction -> Methodology -> Results -> Conclusion）
- [ ] 缺少项目开发过程中的实际踩坑记录
- [ ] 没有任何个人观点或经验分享
- [ ] 代码过于"教科书式"，缺少工程化细节

### 个性化元素检测（有这些可从轻判定）
- [ ] 有具体的踩坑记录
- [ ] 有开发过程中的问题和解决方案
- [ ] 有个人的心得体会和经验总结
- [ ] 代码注释有个性化内容

**判定结果：** 明显AI生成 / 疑似AI生成 / 部分AI辅助 / 无明显AI痕迹

## 第六步：生成评审回复（2-3分钟）

### 回复结构
1. 开头："我看了一下你这个《[项目名]》……"
2. 优点：2-3个具体细节
3. 问题：指出最关键的1-2个卡点
4. 建议：给出具体的修改动作和参考链接
5. 结尾："改完直接私信我，我优先看。"

### 选择回复模板
- [ ] 建议直接加精 → 使用 templates/回复模板库.md 的"一、建议直接加精"
- [ ] 改一版能加精 → 使用 templates/回复模板库.md 的"二、改一版能加精"
- [ ] 这个先放着 → 使用 templates/回复模板库.md 的"三、这个先放着"

### 检查回复风格
- [ ] 接地气的口语，不是机器人
- [ ] 具体的评价，不是空话
- [ ] 可执行的建议，有参考链接
- [ ] 避免大段emoji和夸张赞美
- [ ] 避免机械的编号列表
- [ ] 鼓励性结尾

## 异常处理流程

### 项目链接失效
```
提示：你给的项目链接好像有问题，我访问不了。
建议：检查链接是否正确，项目是否公开，是否已删除
```

### 敏感内容
```
处理：直接拒绝评审
说明：不能包含涉政、暴力、色情等内容
```

### 抄袭/洗稿
```
判定：重复度超过80%
处理：告知不予评审，建议大幅修改或重做
```

### 非飞桨项目
```
判定：未使用飞桨或文心大模型
建议：迁移到飞桨框架或重做
```

## 评审效率提示

### 快速模式（quick focus）
- 只检查：是否公开、是否有飞桨、README是否有内容、是否有代码、是否有依赖文件
- 不进行详细评分
- 直接给出关键问题和修改建议

### 完整模式（full focus）
- 走完完整的七步流程
- 进行详细的五维度评分
- 生成完整的评审回复

### 针对性评审（specific focus）
- 只针对特定问题进行评审（如只看文档质量、只看可复现性）
- 其他维度快速带过
