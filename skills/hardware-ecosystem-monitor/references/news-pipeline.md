# News Pipeline

当前项目中，新闻自动化能力可能来自目标工作区已有的抓取脚本与定时任务；但**新闻是否相关、摘要如何提炼、异常数据是否保留**，应由 Skill 自身完成判断，而不是依赖固定 automation 全量决定。

## 相关文件的维护位置

以下路径描述的是**目标工作区/看板项目中的典型维护位置**，不是本 Skill 仓库随 PR 一起交付的文件：

- 抓取脚本：`scripts/fetch-news.js`
- 输出文件：`public/auto-news.json`
- 工作流：`.github/workflows/schedule-news-fetch.yml`

如果目标工作区没有这些文件，Skill 不应假定“自动抓取已存在”，而应：

1. 明确告知用户该能力当前缺失
2. 回退到手动整理新闻数据
3. 或提出后续可补充 automation 的建议

## 推荐职责分工

### automation 负责

- 定时触发
- 多源采集（RSS / HTML / API）
- 原始数据落盘
- 基础去重与缓存

### Skill 负责

- 判断新闻是否真正属于“大模型 × 多硬件生态”范围
- 根据标题、正文和上下文提炼摘要
- 处理字段缺失、口径冲突和脏数据
- 决定应直接入库、标记待确认还是丢弃
- 在脚本、工作流或密钥缺失时给出替代方案

## 对现有项目的读取方式

如果目标工作区已经存在 `fetch-news.js` 或类似脚本，Skill 应重点检查：

1. 数据源覆盖范围是否仍符合业务目标
2. 输出字段是否匹配前端消费结构
3. 失败回退逻辑是否会产生错误摘要或空字段
4. 定时任务是否真实存在于目标工作区，而不是只写在说明文档里

## 关于外部 AI API

目标工作区里如果已经实现了 Gemini、GLM 或其他外部 AI API 调用，这些都应视为**目标项目自身的实现细节**，而不是本 Skill 的前置依赖。

本 Skill 默认应由其自身完成新闻相关性判断和摘要提炼。只有当用户明确要求“沿用现有项目里的外部 AI 清洗链路”时，才需要检查这些 API 配置是否可用，并说明对应风险与依赖。
