# eco-partner-registration

## 适用场景

当用户需要将一个产品或合作伙伴纳入飞桨/文心生态体系时，使用本 Skill 执行完整的入库提报流程。

**典型触发场景：**
- 用户提供了一款产品介绍，需要提交生态产品认证表格
- 需要生成伙伴协议草案并收集对方工商信息
- 同时完成"产品提报 + 协议提报"两件事
- 需要整理归档提报材料文件夹

**流程说明：**
本 Skill 包含两个阶段，用户可选择只做其中一个，或两个都做（默认两个都做）。执行前先向用户确认执行范围。

**关键约束：**
- 所有模板文件均存放在工作区的 `模板` 文件夹中，AI 需读取后复制使用
- 无法确定的字段一律留空并标注 `[待补充]`，不得猜填
- 涉及公司名称、协议期限等关键信息，必须经用户确认后才能定稿

---

## 输入要求

- **业务目标**：用户要提报的产品/伙伴名称及基本介绍
- **背景上下文**：产品功能描述、技术架构、应用场景（可以是文档/PPT/图片/聊天记录等任意形态）
- **关键约束**：是否指定生态归属（飞桨/文心）；只做产品提报、只做协议提报，还是两者都做
- **期望输出**：填好的认证表格、协议草案、提报邮件草稿、申请理由文本、归档文件夹

---

## 前置检查（执行任何步骤前必须完成）

> ⚠️ 本 Skill 依赖本地模板文件，**这些模板不随 Skill 公开分发**，需使用者自行准备并放置到工作区的 `模板/` 文件夹中。

在开始执行前，先验证以下六个模板文件是否存在：

```bash
ls 模板/
```

需要确认存在的文件：

| 文件名 | 用途 |
|--------|------|
| `飞桨生态产品认证表格模板.docx` | 阶段一 Step 3 使用（归属 = 飞桨时） |
| `文心生态产品认证表格模板.docx` | 阶段一 Step 3 使用（归属 = 文心时） |
| `产品认证提报邮件模板.docx` | 阶段一 Step 4 使用 |
| `申请理由模板.docx` | 阶段二 Step 7 使用 |
| `飞桨伙伴协议模板.docx` | 阶段二 Step 5 使用（归属 = 飞桨时） |
| `文心伙伴协议模板.docx` | 阶段二 Step 5 使用（归属 = 文心时） |

**若文件缺失**：立即停止流程，告知用户："`模板/` 文件夹中缺少以下文件：[列出缺失文件名]，请补充后再继续。"

## 执行步骤

### 阶段一：生态产品提报

#### Step 1：确认执行范围
询问用户：本次执行哪些阶段（仅产品提报 / 仅协议提报 / 两者都做）。

#### Step 2：判断生态归属
- 根据用户提供的产品介绍，判断属于 **飞桨生态产品** 还是 **文心生态产品**
- 不确定时主动询问用户，明确标注：`归属 = 飞桨 / 文心`

#### Step 3：填写产品认证表格
1. 根据 Step 2 确认的归属，从 `模板/` 文件夹选取对应模板并复制重命名：
   - 归属 = 飞桨 → 复制 `飞桨生态产品认证表格模板.docx`，重命名为 `{应用名}-飞桨生态AI产品说明.docx`
   - 归属 = 文心 → 复制 `文心生态产品认证表格模板.docx`，重命名为 `{应用名}-文心生态AI产品说明.docx`
   - 示例：`智能翻译助手-飞桨生态AI产品说明.docx`
2. **编辑方式**：使用 `docx` Skill（`use_skill: docx`）读取模板文件，**先扫描模板中所有需要填写的空缺位置**（包括空白格、占位符、下划线填写区等），再根据用户提供的材料智能提取对应信息逐一填入：
   - 对每个空缺字段，从用户材料中主动推断最匹配的内容
   - 无法从材料中推断的字段，保留空白并标注 `[待补充: 字段名]`
   - 不得自行捏造或假设任何信息
3. 填写完成后，向用户输出两项内容：① 已填写字段的摘要确认；② 所有 `[待补充]` 字段的清单，逐一说明需要用户补充什么信息。循环修改直到用户回复"确认"

#### Step 4：生成提报邮件正文
1. 从 `模板/` 文件夹找到 `产品认证提报邮件模板.docx`
2. 根据定稿表格内容生成邮件正文
3. 收件人、抄送人留空，并提示用户：**"请自行填写收件人/抄送人，注意抄送人排序规则"**
4. 将邮件草稿归档到对应产品文件夹

---

### 阶段二：伙伴协议提报

#### Step 5：填写伙伴协议
1. 根据生态归属，从 `模板/` 文件夹选取对应模板并复制重命名：
   - 归属 = 飞桨 → 复制 `飞桨伙伴协议模板.docx`
   - 归属 = 文心 → 复制 `文心伙伴协议模板.docx`
   - 重命名为：`{伙伴名称}-{协议类型}-协议草案.docx`
2. **编辑方式**：使用 `docx` Skill（`use_skill: docx`）读取模板文件，**先扫描协议中所有需要填写的变量位置**（包括空白格、「」填写区、下划线等占位形式），再根据产品说明和已收集的外部信息智能填入：
   - 对每个变量位置，从已有信息中主动推断最匹配的内容
   - 无法确定的变量保留空白并标注 `[待补充: 字段名]`
   - **期限字段**严格按下方规则计算，不得推断
   - 不得自行捏造或假设任何信息
3. **期限规则**（重要）：
   - 开始日期 = 提报当月最后一天
   - 结束日期 = 次年对应日期的前一天
4. 归档到 `{伙伴名称}_协议材料/` 文件夹

#### Step 6：外部信息收集与自查
1. 基于产品名称、伙伴名称，联网搜索：公司全名、注册地址、官网、工商信息等基础公开数据
2. 主动询问用户：对方负责人姓名、对接人姓名
3. 将收集到的信息汇总为 checklist，发给用户确认无误

#### Step 7：生成申请理由与最终提醒
1. 从 `模板/` 文件夹找到 `申请理由模板`，结合产品认证表格和外部信息生成一段申请理由
2. 同步生成**提醒事项列表**，包括但不限于：邮件抄送人排序、[待补充] 字段清单等
3. 等待用户确认后，将所有生成文件（认证表格、协议草案、申请理由）统一放入新建文件夹：
   - 命名规则：`{日期}_{产品名}_提报材料`
   - 示例：`20260422_智能翻译助手_提报材料`

## 输出要求

- **结构清晰**：每个文件单独输出，明确文件名和归档路径
- **结论与过程一致**：填写内容需对应用户提供的原始材料
- **明确风险与假设**：
  - 所有 `[待补充]` 字段在交付时需单独汇总告知用户
  - 工商信息来源于公开搜索，需用户二次确认


---

## 踩坑经验

- **归属判断不确定时一定要问**，不要自行假设为飞桨或文心，归属错误会导致整个提报流程返工
- **期限计算**：开始日期是"当月最后一天"而非"今天"，结束日期是"次年前一天"，容易写错。
- **模板文件路径**：所有模板在 `模板` 文件夹下，读取前确认文件存在，若找不到需提示用户检查路径
- **邮件抄送人排序**：这是用户反复强调的注意点，生成邮件时务必提醒，不要遗漏此提示
