# 记忆持久化完整规则

## 为什么记忆持久化是核心问题

交接不是一次性的文档整理——它要求 Agent 在交接完成后，仍能像原负责人一样回答问题。这意味着知识必须从"会话内存"转移到"持久化存储"，且在新会话中可被恢复。

本文件总结了记忆持久化的完整规则、常见陷阱和最佳实践。

---

## 1. 写入规则

### 1.1 必须显式写入

**规则**：在会话中必须使用 Write/Edit 工具将知识写入文件。不要依赖会话结束时的自动保存。

**原因**：Agent 的会话内存在对话结束后会被清除。只有写入磁盘文件的知识才能跨会话保留。

**正确做法**：
```
Step 6 完成知识提炼后，立即执行 Write 将内容写入 handover-[项目名].md
```

**错误做法**：
```
在对话中说"我已经记住了"——这不会持久化
```

### 1.2 写入后必须验证

**规则**：每次 Write 操作后，立即使用 Read 工具读取刚写入的文件，确认内容完整无误。

**原因**：Write 操作可能在以下场景静默失败：
- 路径错误（写到了不存在的目录）
- 权限不足
- 磁盘空间不足
- 文件被其他进程锁定

**验证标准**：
- 文件存在且可读
- 关键信息（流程步骤、联系人、FAQ）完整
- 格式正确（Markdown 渲染正常）

### 1.3 更新 MEMORY.md 索引

**规则**：每次新增交接记忆文件后，必须在 MEMORY.md 中追加索引条目。

**原因**：新会话启动时，Agent 只会读取 MEMORY.md。如果新文件没有在索引中，新会话不会知道它的存在。

**索引格式**：
```markdown
## 交接知识

| 交接项目 | 交接人 | 覆盖维度 | 记忆文件 | 日期 |
|---------|--------|---------|---------|------|
| [项目名] | [名称] | 6/6 | [handover-xxx.md](handover-xxx.md) | [日期] |
```

## 2. 读取规则

### 2.1 新会话启动时的自动发现

**规则**：新会话启动后，如果用户的问题涉及某个项目/业务，Agent 应该：
1. 读取 MEMORY.md
2. 检查"交接知识"索引
3. 如果有相关交接记忆文件，主动 Read 该文件
4. 基于记忆内容回答问题

### 2.2 回答时标注来源

**规则**：基于交接记忆回答问题时，必须标注知识来源。

**标注格式**：
```
根据交接资料（来源：[文件名]）：
[回答内容]
```

### 2.3 不知道就说不知道

**规则**：如果交接资料中没有覆盖用户的问题，诚实回答"交接资料中未覆盖此信息"，不要用训练语料中的通用知识冒充交接知识。

**正确做法**：
```
交接资料中未覆盖"XX流程的历史决策背景"。
建议向原负责人 [姓名] 确认，或补充交接材料后更新记忆。
```

**错误做法**：
```
基于行业经验，XX流程的历史决策背景是...（这是编造的，不是交接知识）
```

## 3. 更新规则

### 3.1 增量更新

**规则**：交接完成后如果获得新信息（如向原负责人追问了某个问题），应该增量更新记忆文件，而不是重写整个文件。

**操作**：
1. Read 现有记忆文件
2. 将新信息追加到对应维度
3. 更新覆盖率（如有变化）
4. Write 更新后的文件

### 3.2 定期归档

**规则**：如果交接知识超过 6 个月未更新，应该在索引中标注"可能过时"，提醒用户确认是否需要更新。

**索引标注**：
```markdown
| [项目名] | [名称] | 6/6 | [handover-xxx.md](handover-xxx.md) | 2026-04-22 ⚠️ 超6个月未更新 |
```

### 3.3 冲突处理

**规则**：如果新信息与已有记忆矛盾：
1. 保留矛盾，标注 `[更新]` 和日期
2. 说明矛盾的原因（如流程变更、系统升级）
3. 新信息标记为"最新"，旧信息标记为"历史版本"

## 4. 常见陷阱

### 陷阱 1：以为会话记忆会持久

**现象**：在对话中处理了大量交接信息，以为 Agent 会自动记住。

**真相**：会话结束后，所有未写入文件的信息都会丢失。

**预防**：每个 Step 完成后立即写入文件，不要等到最后。

### 陷阱 2：多会话之间不互通

**现象**：在会话 A 中写了交接记忆，在会话 B 中问相关问题却回答不上来。

**真相**：每个会话是独立的，不会自动读取其他会话的文件。

**预防**：维护 MEMORY.md 索引，新会话启动时主动检查索引。

### 陷阱 3：记忆文件放在 Skill 目录内

**现象**：将交接记忆写入 Skill 的 references/ 目录。

**问题**：Skill 是可以安装/卸载的。卸载 Skill 时记忆也丢了。

**预防**：交接记忆应该放在记忆目录（如 `.claude/memory/`），不在 Skill 目录内。

### 陷阱 4：FAQ 编造

**现象**：交接材料不够，Agent 自己编造 FAQ 来"凑数"。

**问题**：编造的 FAQ 在使用时会被识破，导致用户对 Agent 失去信任。

**预防**：FAQ 必须基于真实的交接材料。未覆盖的内容标记为 `[待补充]`。

### 陷阱 5：记忆文件过于冗长

**现象**：将完整的交接手册内容全部写入记忆文件。

**问题**：文件过长会浪费上下文窗口，且大部分细节并不常用。

**预防**：记忆文件只保留"快速参考"级别的内容（核心流程、关键联系人、FAQ、风险备忘），详细内容引用交接手册原文。

---

## 5. 记忆文件大小控制

| 文件类型 | 建议大小 | 内容范围 |
|---------|---------|---------|
| MEMORY.md 索引 | < 200 行 | 所有索引条目 |
| 交接专项记忆 | < 500 行 | 快速参考 + 核心知识摘要 |
| 交接手册 | 无限制 | 完整知识（详细参考用） |

**原则**：记忆文件是"索引+摘要"，交接手册是"完整文档"。两者配合使用，不是互相替代。
