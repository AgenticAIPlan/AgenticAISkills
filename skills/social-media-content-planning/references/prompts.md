# Prompt 设计说明

本 Skill 涉及三类 prompt：**Topic Plan**、**Draft**、**Review**。三者互相校验，形成闭环。

---

## 1. Topic Plan（选题规划）

**用途**：让模型从素材中发散出 3 个可写角度，避免"一眼定稿"带来的同质化。

**关键设计**：
- 在 system prompt 里固定品牌调性（audience / language / style / voice），不随每次请求变化
- 要求模型为每个角度单独给出"为什么适合这批受众"的论证，强制它对读者负责
- 温度 0.7，兼顾多样性与相关性

**调优建议**：
- 若角度重复度高 → 明确要求"3 个角度必须分别偏向：技术向 / 产品向 / 生态向"
- 若角度过于空泛 → 要求每个角度必须引用素材中的 1 个具体事实锚点

---

## 2. Draft（草稿撰写）

**用途**：从 3 个角度里选出最优 2 个，产出完整推文。

**关键设计**：
- 强制 `[DRAFT A]` / `[DRAFT B]` 标签，方便下游脚本或审稿模型定位
- 要求包含 hashtags 与推荐发布时间（对分散地域受众尤其必要）
- 温度 0.8，鼓励文案层面的创造性
- 单条 ≤ 280 字符 / Thread 每条 ≤ 280 字符 — 平台硬约束，放在 system prompt 里反复强调

**调优建议**：
- 若草稿过于营销 → 在 CONTENT GUIDELINES 里加一条"禁止使用 'revolutionary' / 'game-changer' / 'unprecedented' 等超级词"
- 若草稿过于干瘪 → 要求"至少包含一个可验证的数字或具体场景"

---

## 3. Review（独立审稿）

**这是本 Skill 的核心防线**。使用 **不同厂商或不同代次的模型**，避免"同一个模型自评自赞"的失效场景。

**六维检查**：

| 维度 | 检查重点 | 常见失效 |
|------|---------|---------|
| 语气 | 是否适合开发者受众 | 过度营销、官腔 |
| 歧义 | 句子是否可能被误读 | 代词指向不明、省略主语 |
| 绝对化 | "the best / only / always" 等 | 削弱可信度、易被反驳 |
| 事实准确性 | 对照素材逐条核查数据 | 运营成本 ≠ 前期投入、1/10 vs 1/3 |
| 文化适配 | 本地化表达 | 中式英语、文化梗失效 |
| 平台适配 | 字符数 / Thread 结构 | 单条超 280、Thread 编号断裂 |

**输出格式要求**：
- 每条草稿必须给出 `READY / NEEDS MINOR EDITS / NEEDS REWRITE` 三档结论
- 问题必须**引用原文 + 给出修改建议**，不能停留在"有点自夸"这种模糊评价
- 最后附 Revised Version，让运营看到"改完长什么样"

**温度 0.3**：审稿追求稳定与一致性，不要创造性。

---

## 模型搭配建议

| 写稿模型 | 推荐审稿模型 | 理由 |
|---------|------------|------|
| GLM-4.7 | GLM-5 | 同家族不同代次，既理解风格又能看出老模型的"套路" |
| GPT-4o | Claude Sonnet | 跨厂商，审稿视角差异大 |
| DeepSeek-R1 | Llama-3.3-70B | 推理模型 + 对话模型，审稿更快 |

**不推荐**：用同一个模型写稿 + 审稿。实测会出现"模型自认为写得很好"的系统性偏差。
