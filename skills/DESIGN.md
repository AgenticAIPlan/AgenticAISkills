# 飞桨星河项目评审专家Skill - 设计文档

## 一、快速筛选的具体标准

### 1.1 水文判定标准
**以下情况可直接判定为水文，建议告知问题后跳过评分：**
- README内容少于50字，且无实质内容
- 只有README，没有任何代码文件
- 代码文件数量少于3个，且总代码行数少于100行（排除注释和空行）
- 项目标题与内容完全不相关
- 纯复制粘贴官方示例代码，无任何修改或说明
- 明显的广告、引流内容

### 1.2 环境配置检查清单
**检查顺序和标准：**
1. 是否有依赖文件（requirements.txt / setup.sh / environment.yml / environment.yaml）
2. README中是否有环境安装说明
3. 是否指定了飞桨版本号
4. 数据集路径是否使用相对路径或正确配置（如 `/home/aistudio/data/`）

## 二、AI生成内容判定标准与减分规则

### 2.1 AI生成内容特征
**高疑似AI生成特征（任一即触发）：**
- README完全通用模板化，缺乏具体项目细节（如"本项目展示了...的强大能力"这类万能句式）
- 代码注释高度统一、格式过于完美，缺乏人类书写时的个性化
- 代码结构和变量命名过于规整，缺少实际业务场景的痕迹
- 文档中充斥"全面"、"强大"、"高效"等大词，但缺乏具体数据支撑
- 多处出现AI常见表达："在本文中"、"值得注意的是"、"总之"等

**中等疑似AI生成特征（需多方面综合判断）：**
- README结构过于标准：Introduction -> Methodology -> Results -> Conclusion
- 缺少项目开发过程中的实际踩坑记录和解决方案
- 没有任何个人观点或经验分享
- 代码过于"教科书式"，缺少实际工程化的细节处理

### 2.2 检测维度
1. **README内容检测**：通用模板化程度、具体信息含量
2. **代码注释检测**：统一度、实用性、个性化程度
3. **文档风格检测**：词汇选择、句式结构、语气一致性
4. **个性化元素检测**：是否包含实际调试过程、遇到的问题、个人经验

### 2.3 AI生成判定结果与减分档位

| 判定结果 | 特征描述 | 减分规则 |
|---------|---------|---------|
| **明显AI生成** | 高疑似特征≥3个，且无任何个人化内容 | 创新性-10分，技术深度-5分，文档质量-5分（共-20分） |
| **疑似AI生成** | 高疑似特征1-2个，或中等疑似特征≥3个 | 创新性-5分，技术深度-3分（共-8分） |
| **部分AI辅助** | 有AI生成痕迹，但有明显的个人修改和补充 | 创新性-3分，文档质量-2分（共-5分） |
| **无明显AI痕迹** | 无上述特征，或AI仅用于翻译/格式化等辅助用途 | 不减分 |

**特别说明：**
- 如果项目标注了"AI辅助生成"且提供了详细的人工修改说明，可适当从轻判定
- 如果完全AI生成但代码质量和文档质量极高，且确实对社区有价值，可酌情考虑"改一版能加精"，要求增加个人心得和经验总结
- AI生成内容如果涉及抄袭，按抄袭处理，不考虑加精

## 四、可复现性检测要点

### 4.1 数据集公开性检查
- 数据集是否来自公开来源（Kaggle、UCI、百度AI Studio数据集等）
- 是否提供了数据集的下载链接或获取方式
- 如果是个人数据，是否说明数据格式和字段含义

### 4.2 依赖完整性检查
- **必需文件**：requirements.txt / setup.sh / environment.yml至少一个
- **关键依赖**：paddlepaddle版本必须明确指定
- **可选依赖**：可视化工具、数据处理库等

### 4.3 运行说明检查
- 环境安装命令是否清晰
- 数据集准备步骤是否完整
- 模型训练/推理命令是否明确
- 预期运行时间和资源消耗是否说明

## 七、异常处理流程

### 7.1 项目链接失效
- 提示用户检查链接是否正确
- 如果是私有项目，提示用户改为公开或直接提供项目内容
- 如果项目已删除，询问是否有备份

### 7.2 敏感内容处理
- 涉及政治、暴力、色情等内容：直接拒绝评审
- 涉及商业机密或未授权数据：提示用户脱敏后再提交
- 涉及恶意代码或攻击性内容：拒绝评审

### 7.3 抄袭/洗稿处理
- 与已有项目代码重复度超过80%：直接告知不予评审
- 明显复制粘贴但改了变量名：告知需要大幅修改，建议参考"如何正确使用开源代码"
- 有引用说明但无原创内容：建议增加自己的理解和改进

### 7.4 AI生成内容处理
- 完全AI生成且无标注：告知这是AI生成，需要补充个人理解和修改
- AI生成但标注清楚且做了大量修改：按"部分AI辅助"处理
- 如果开发者主动说明用了AI辅助，给予正向引导

## 十、工作流程补充（AI生成检测步骤）

### 第四步：AI生成内容检测（新增）

在"心里打分"之前，先快速检测是否有AI生成痕迹：

1. 扫描README，检查是否有明显的AI生成特征
2. 检查代码注释的统一度和实用性
3. 判断是否有个人化元素（踩坑记录、心得体会）
4. 根据特征数量判定AI生成程度（明显/疑似/部分/无）
5. 在心里打分时，根据AI生成程度在相应维度减分

### 第五步：综合评分和判定

1. 按五个维度打分，考虑AI生成减分
2. 计算总分
3. 根据总分和关键问题，给出三类结论之一
4. 在回复中，如果检测到AI生成，委婉指出并给出改进建议

## 十一、加精底线更新

1. 公开项目，用的必须是飞桨或者文心大模型的技术。
2. README清晰易懂，不能只有很少的内容。
3. 代码要在 AI Studio 环境里跑得通。
4. 明显灌水或者洗稿的直接忽略。
5. **完全AI生成且无个人化内容的项目不建议加精**（新增）。
