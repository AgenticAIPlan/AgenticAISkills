# Review Rubric

## 仓库规范

- 确认 PR 目标分支是 `dev`，而不是 `main`
- 确认分支名符合 `feat/<真实姓名拼音>/<skill-slug>` 或 `update/<真实姓名拼音>/<skill-slug>`
- 确认改动主要落在单个 `skills/<skill-slug>/` 目录
- 对直接修改 `skills/_template/`、`skills/README.md`、仓库级插件元数据的业务同学 PR 保持高警惕
- 如果 PR 描述里能看到真实姓名拼音、Skill 路径、Agent 提交字段，顺手检查是否自洽

## Skill 结构

- `skills/<skill-slug>/SKILL.md` 必须存在
- frontmatter 只应包含 `name` 和 `description`
- `name`、目录名、PR 中的 Skill 名称应大体一致，不要出现明显错位
- `description` 应同时说明 Skill 做什么，以及什么情况下触发
- `references/`、`assets/`、脚本都是可选项；只有在确实提升复用性时再要求
- 不要因为缺少 `agents/openai.yaml`、示例文件或额外文档就直接打回

## Skill 质量

- 描述是否足够具体，能让 Agent 知道何时使用这个 Skill
- 输入要求是否清楚，是否说明任务边界、必要上下文和关键约束
- 执行步骤是否可落地，是否避免“自行发挥”“按需处理”这类空话
- 输出要求是否可检查，是否说明产物结构、结论形式、或需要显式标注的风险和假设
- 内容是否足够精炼，避免把常识解释得过长
- 长篇细节是否放进 `references/`，避免把 `SKILL.md` 写成杂糅手册
- 如果加入脚本，确认脚本确实服务于重复执行或稳定性，而不是徒增复杂度

## 结论分级

- `必须修改`：缺少 `SKILL.md`、frontmatter 错误、触发条件缺失、步骤不可执行、输出要求缺失到无法使用、明显违反仓库规则
- `建议优化`：结构基本成立，但触发描述、步骤颗粒度、输出格式、上下文组织还可以更稳
- `可选增强`：不影响合入的增强项，例如补充一两个参考资料、压缩冗余措辞、增加更明确的示例

## 评论格式

评论首行固定写：

`【Code X Agent 自动评论｜自动扫描】`

评论正文建议按下面的顺序组织：

1. 一句话结论
2. 1 到 2 个亮点
3. `必须修改` 或 `建议优化`
4. 可选增强

如果整体质量已经不错，就直接给出正向结论，不要为了显得严格而强行挑刺。
