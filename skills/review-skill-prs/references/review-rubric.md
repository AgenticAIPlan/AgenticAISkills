# Review Rubric

## 一、业务成立性

先判断这个 Skill 值不值得存在，再判断它写得好不好。

### 业务五问

- 这是不是一个稳定重复出现的业务任务，而不是一次性请求。
- 使用者、触发时机、上游输入、下游交付是不是清楚。
- 任务里是否包含真实的业务判断，而不是纯格式加工。
- 输出是否能直接服务下一步业务动作。
- 风险、假设、需要人工确认的事项是否被显式写出。

### 业务侧常见问题

- 场景过宽：几乎任何部门都能套用。
- 任务过虚：没有明确动作，只有“分析、优化、生成”。
- 业务判断缺失：没有优先级、例外、红线、升级路径。
- 输出不可验收：只有“给出建议”或“整理结果”，没有结构和完成标准。
- 风险缺失：明明是高风险场景，却不要求标注不确定项或人工确认。

### 业务侧结论

- `必须修改`：业务场景不真实、过于空泛、没有明确产物、缺少关键业务判断或风险边界。
- `建议优化`：业务场景成立，但输入、例外、输出或风险说明还不够稳。
- `可选增强`：补充少量业务示例、反例、升级条件或验收口径。

## 二、Skill 设计质量

### Trigger 质量

- `description` 是否同时覆盖“做什么”和“什么时候用”
- 是否能和相邻 Skill 区分开，避免抢触发
- 是否只是标题换写，没有真正的触发条件

### Workflow 质量

- 输入要求是否完整且现实
- 步骤是否可执行，而不是空话
- 输出要求是否可检查
- 是否说明失败处理、边界和不确定项

### Progressive Disclosure

- `SKILL.md` 是否只保留核心流程和选择规则
- 长篇知识、示例、细则是否下沉到 `references/`
- 如果只靠一个超长 `SKILL.md` 堆满内容，是否真的必要

### Resource 设计

- 重复性强、容易写错、需要稳定性的动作，是否应该抽成脚本
- 需要长期引用的领域资料，是否应该放在 `references/`
- 不要为了“看起来完整”而硬塞可选目录

### 一个重要判断

`SKILL.md` 只有几百行不是问题本身。

真正的问题是：如果整个 Skill 只有一份主文件，而且里面既没有业务判断，也没有边界、失败处理、参考资源、脚本或清晰的输出约束，那它更像一段长提示词，不像可复用 Skill。

## 三、仓库规范

- PR 目标分支是否正确
- 分支命名、Skill 路径、PR 模板字段是否自洽
- 是否只改了一个 `skills/<skill-slug>/`
- 是否错误修改模板、目录说明或仓库级元数据

仓库规范问题通常不应掩盖业务和 Skill 质量问题，两者都要写。

## 四、评论模板

评论首行固定：

`【Code X Agent 自动评论｜自动扫描】`

推荐结构：

1. 一句话结论
2. 1 到 2 个亮点
3. `必须修改` 或 `建议优化`
4. 可选增强

如果要更直接，可以用这些结论句：

- `业务场景还不够实，建议先补强实际任务边界后再合入。`
- `结构基本成立，但目前更像长提示词，还没完全长成可复用 Skill。`
- `业务和 Skill 两侧都已经过线，小修后可合入。`
