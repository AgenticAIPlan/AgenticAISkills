# 任务形态

本文件只回答输入是什么形态、基础 OCR 链路是否需要检测，以及是否需要方向或矫正类可选模块。

本文件不负责选择部署拓扑、目标平台、接入宿主，也不展开运行时路线。

## 1. 基础判断

- **整页图、拍照图、海报、票据、文档照片：**默认从“检测 + 识别”的基础 OCR 链路开始判断。
- **已经裁切好的单行或单词图：**可以先按“仅识别”处理，但 PoC 中仍要区分检测误差和识别误差，避免把上游裁切问题误判成模型问题。

## 2. 可选模块

只在输入特征明确需要时再加模块：

- **文档方向分类：**整页图像整体旋转明显时考虑。
- **文本图像矫正：**曲面页、强透视、弯折文档照片明显时考虑。
- **文本行方向分类：**检测出的文本行存在竖排、倒置或方向混杂时考虑。

不要在没有样本证据时默认接入这些模块。它们会改变链路复杂度，也可能引入新的误差。

## 3. 模型变体

`mobile` 和 `server` 是公开模型表里的变体名称，不是部署位置。

- **`mobile`：**通常更适合延迟、内存、包体更紧的场景。
- **`server`：**通常更适合设备预算允许、且更重视质量的场景。

在回答中把“任务形态”和“模型变体”分开写，不要用“端侧就一定 mobile”这种说法。

如果字符集或语种范围会影响识别模型选择，应在输出中单列为待确认项。本文件不展开具体语种模型目录。

## 4. 快速判断

- 如果用户要“照片直接出文本”，默认从“检测 + 识别”的基础链路开始。
- 如果用户已经有稳定裁切结果，才把“仅识别”当成主路线。
- 如果用户只说“精度不够”，先看是不是漏检、方向错、透视畸变，而不是直接换更大模型。

## 5. 反模式

- 不要把 `mobile` / `server` 当成“手机版 / 服务器版”。
- 不要把文档方向分类、文本行方向分类和文本图像矫正当成默认必选项。
- 不要在还没确认输入形态前就跳到运行时或导出问题。
