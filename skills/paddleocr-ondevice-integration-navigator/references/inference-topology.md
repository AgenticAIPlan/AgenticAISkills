# 部署拓扑

本文件只回答 OCR 链路应部署在端上、云上，还是端云协同，以及端上与云上的职责如何拆分。

本文件不负责判断目标平台、接入宿主形态，也不负责选择具体运行时或导出路径。

## 1. 先判断是不是端侧相关问题

这份技能服务的是“端侧相关的集成判断”，不是“任何 OCR 服务架构”。

- **纯端侧：**检测、识别和必要可选模块都在设备侧完成。
- **端云混合：**端上负责一部分链路，云上负责另一部分链路。
- **纯云：**OCR 主链路在云上完成，端上主要承担采集、上传、交互、缓存或重试等辅助职责。

如果用户的问题已经退化成“普通服务器 OCR 部署”，应退出本技能范围。

## 2. 什么时候优先纯端侧

以下场景优先从纯端侧开始判断：

- 强离线要求。
- 图像或文本不允许上传。
- 网络时延或可用性不稳定。
- 设备性能足以支撑目标模型变体和目标时延。

纯端侧的代价通常是：

- 包体、内存、端上算力压力更大。
- 模型更新、灰度和端上兼容性更复杂。

## 3. 什么时候优先端云混合

以下场景通常值得优先考虑端云混合：

- 端上有一定约束，但不要求完整离线。
- 需要兼顾隐私、交互体验和更高质量的云端能力。
- 端上只能承担轻量步骤，无法稳定跑完整 OCR 链路。
- 浏览器或跨端宿主场景中，纯端侧路线往往需要更严格的可行性验证。

端云混合不是一句“部分放云上”就结束，必须继续追问：

- 哪些数据允许上传。
- 哪些环节必须端上完成。
- 弱网或断网时允许退化到什么程度。
- 用户要的是实时交互还是异步处理。

## 4. 什么时候采用纯云

以下场景通常更接近纯云：

- 识别主链路可以稳定放到云上。
- 端上主要负责采集、压缩、脱敏、上传、展示结果或失败重试。
- 用户真正关心的是端侧交互约束，而不是把完整 OCR 链路留在端上。

如果纯云方案仍受端侧工程约束影响，应在输出中单列说明这些约束，例如：

- 本地采集格式是否适合上传。
- 上传前是否需要脱敏、压缩、裁切或缓存。
- 网络失败时是否需要排队重试、延迟提交或本地占位结果。

此时应明确告诉用户：核心 OCR 能力并不在端上，但端侧工程约束仍决定集成方案。

## 5. 端云混合的常见拆分方式

可以优先从这几种拆分方式中选择：

### A. 端上采集与预处理，云上完整 OCR

适合：

- 端上只需要做拍照、裁切、压缩、基础增强。
- 云上可以承担完整检测、识别和后处理。

优点：

- 端上最轻。
- 云上模型和流程更容易统一升级。

缺点：

- 对网络和上传链路依赖最高。
- 隐私与时延压力更大。

### B. 端上检测 / 裁切，云上识别或后处理

适合：

- 端上希望减少上传范围，或先做局部裁切。
- 用户更在意交互反馈和带宽，而不是完全离线。

优点：

- 可减少上传的数据量。
- 端上可更快给出框选或局部反馈。

缺点：

- 端云之间必须严格对齐裁切结果、颜色空间和坐标系。
- 链路拆分更复杂，责任边界更容易模糊。

### C. 端上轻量 OCR，云上兜底重模型或业务后处理

适合：

- 端上需要即时反馈。
- 云上只在低置信度、复杂样本或联网条件允许时介入。

优点：

- 兼顾端上体验和复杂场景质量。
- 可把云端调用收敛到真正困难的样本。

缺点：

- 需要明确触发云端兜底的条件。
- 一致性、回退策略和监控更复杂。

## 6. 输出时必须说清楚的内容

一旦推荐端云混合，或推荐纯云但端侧仍存在明显工程约束，输出里必须单列：

- 端上负责什么。
- 云上负责什么。
- 哪些数据会上传。
- 断网或弱网时如何降级。
- 哪些指标分别在端上和云上验证。

不要只写“采用端云混合架构”而不说明拆分方式。
