# 接入形态

本文件只回答 OCR 能力应如何挂载到业务宿主中，例如原生直接集成、跨端桥接或封装原生 SDK。

本文件不负责决定是否采用端云混合，也不负责选择具体运行时或导出格式。

## 1. 原生应用

原生应用场景下，OCR 能力直接集成到 iOS / Android 工程里。

这类问题通常要继续判断：

- 目标平台到底是 iOS、Android，还是双端。
- 是直接在当前工程中接入 OCR 能力，还是抽象成可复用的原生能力。
- 这项能力是否需要对多个页面、多个业务模块或多个应用复用。

默认按下面顺序收敛：

- **需要原生端长期复用、稳定对外暴露能力边界：**优先建议封装成原生 SDK，再由上层业务调用。
- **只是单个页面或单个流程快速接入：**可以先用较薄的原生集成层直接接入，但仍应说明后续可能要演进成 SDK。

要明确告诉用户：PaddleOCR 目前没有现成的原生 iOS / Android 官方集成示例可直接照搬，最终落地仍需要用户自己编写集成代码。

## 2. 跨端宿主框架

React Native、Flutter 等属于跨端宿主框架。

跨端场景下要继续判断：

- 是通过桥接调用原生侧集成代码。
- 还是把 OCR 能力封装成原生 SDK，再由跨端层调用。
- 当前业务更重视快速验证，还是更重视长期复用与宿主层稳定性。

默认按下面规则给出 `推荐接入形态`：

- **只需尽快验证单一业务流程，且团队能维护少量桥接代码：**优先桥接调用原生侧集成代码。
- **需要在多个页面、多个业务模块或多个应用里复用 OCR 能力：**优先把 OCR 能力封装成原生 SDK，再由跨端层调用。

因此，跨端问题通常不会单独给出一条“跨端平台路线”，而是继续回到：

- 目标平台判断。
- 部署拓扑判断。
- 运行时与导出判断。

如果最终仍无法在“桥接 / 原生 SDK”之间明确落位，就说明关键信息还不够，应该继续追问，而不是停在笼统描述。
