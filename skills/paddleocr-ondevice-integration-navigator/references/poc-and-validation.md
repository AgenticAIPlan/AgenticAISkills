# PoC 与验证

## 1. 最低输入要求

开始 PoC 前，至少要书面确认这些字段：

- 数据来源：代表性真实样本，而不只是干净扫描件。
- 主指标：字符 / 词准确率，或明确的业务 KPI，二选一作为主指标。
- 延迟口径：冷启动、热启动、单张、批量分别怎么记。
- 设备信息：SoC、操作系统版本、可用 CPU / GPU / NPU。

## 2. 公平对比

如果要比较云端、边缘端或不同运行时，至少保证：

- 使用同一批输入样本。
- 使用同一套标注与统计口径。
- 明确记录预处理差异，例如缩放上限、颜色空间、二值化、裁切逻辑。

如果对比对象包含云 API，不要只比精度；还要把网络、隐私、时延波动写入非功能维度。

## 3. 失败模式探测

PoC 里建议主动拆开的失败模式：

- **漏检 vs 识别混淆：**用中间裁切图把检测问题和识别问题分开。
- **旋转 / 竖排 / 混合方向：**单独做测试子集。
- **长行 / 极小字：**单独做测试子集，避免平均指标掩盖问题。
- **透视 / 弯折 / 曲面页：**如果场景存在，单独验证是否真的需要矫正模块。

## 4. 数字纪律

延迟、体积、精度等数字只能来自：

- 用户自己的实测。
- 官方模型表。
- 已明确说明来源的内部基准。

没有来源时写 `待验证`，不要补一个“经验值”。

## 5. 通过门槛

回答里最好给出明确通过门槛，而不是只说“先试试”：

- 什么条件算 PoC 通过。
- 什么条件触发换模型变体。
- 什么条件说明问题不在模型，而在工程链路。
