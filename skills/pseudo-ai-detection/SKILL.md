# Pseudo-AI Detection

## 适用场景

当用户已经提出一个 AI 方向的需求，但团队不确定这个需求是不是伪需求、是不是应该现在推进、是不是应该先解决流程或数据问题时，使用本 Skill。

典型触发场景：

- 需求描述听起来像愿景而不像项目
- 业务方说“想做个 AI 系统”，但说不清谁来用、怎么用
- 立项讨论中感觉方向对，但第一阶段不知道怎么验证
- 团队怀疑这个问题更像流程问题、数据治理问题，或当前推动条件不具备

## 输入要求

- 需求名称或一句话需求描述
- 目标用户或使用角色
- 当前人工处理方式
- 想通过 AI 解决的核心问题
- 已知约束：数据、流程、部门协同、风险、上线要求

如果信息不完整，至少要能说明：

- 谁会用
- 现在怎么做
- 为什么现在想做
- 如果做错会出什么问题

## 执行步骤

1. 对照以下八项检查逐一判断：
   - 有没有明确用户
   - 有没有具体任务
   - 需求写得像愿景还是像项目
   - 第一阶段能不能验证
   - 问题更像流程问题还是 AI 问题
   - 问题更像数据治理问题还是 AI 问题
   - 当前是否具备推动条件
   - 做错的业务风险是否过高
2. 根据判断结果分为四类：
   - 继续推进：不是伪需求，可以进入下一步项目定义
   - 缩小重写：方向对，但范围过大或验证目标不清
   - 非 AI 优先解：当前更适合先做流程治理、数据治理或规则代码
   - 暂缓：条件不具备或风险过高，当前不建议推进
3. 如果结论是“缩小重写”，必须把需求重新收敛成：
   - 一个明确用户
   - 一个具体任务
   - 一个第一阶段可验证目标
4. 如果结论是“非 AI 优先解”或“暂缓”，必须明确写出原因，而不是只写“暂时不做”
5. 输出结构化检查结果，并明确下一步动作

## 输出要求

输出必须使用结构化 Markdown，至少包含以下内容：

```markdown
## 伪 AI 需求检查结果

**需求名称：** [填入]
**检查日期：** [填入]

### 1. 八项检查结果
| 检查项 | 判断 | 说明 |
|--------|------|------|
| 1. 没有明确用户 | 是 / 否 | |
| 2. 没有具体任务 | 是 / 否 | |
| 3. 写得像愿景 | 是 / 否 | |
| 4. 第一阶段无法验证 | 是 / 否 | |
| 5. 更像流程问题 | 是 / 否 | |
| 6. 更像数据治理问题 | 是 / 否 | |
| 7. 自己推不动 | 是 / 否 | |
| 8. 做错风险过高 | 是 / 否 | |

### 2. 判断结论
- 结果：继续推进 / 缩小重写 / 非 AI 优先解 / 暂缓
- 一句话理由：[填入]

### 3. 行动建议
- 下一步：[具体动作]

### 4. 如需重写
- 目标用户：[填入]
- 具体任务：[填入]
- 第一版验证目标：[填入]
- 当前不做什么：[填入]
```

输出必须明确说明风险、假设和待确认项，不能只给口号式建议。

## 参考资料

如果已有需求文档、会议纪要、立项材料、现有 SOP 或流程说明，应优先结合这些资料判断，不要只根据一句“想做 AI”就下结论。
