# Agent Boundary Design

## 适用场景

当用户需要为某个业务 Agent 明确以下内容时，使用本 Skill：

- Agent 应该自主完成什么
- 哪些动作必须人工确认
- 哪些高风险动作必须保留给人工
- 单 Agent 是否应该拆成多个 Agent
- 当前 Agent 是否因为“做太多”或“做太少”而影响用户信任

## 输入要求

- Agent 服务的业务目标或任务描述
- Agent 面向的用户角色
- Agent 可能执行的关键动作列表
- 高风险动作、合规约束或错误代价
- 是否存在人工审核、人工接管或多 Agent 协作场景

## 执行步骤

1. 先写清楚该 Agent 的单一职责，使用一句 job story 表达：当什么情况发生时，Agent 需要完成什么动作，以帮助用户达成什么结果。
2. 列出该 Agent 最关键的 3 个高风险动作，并逐项判断：
   - 是否可逆
   - 一旦出错代价有多高
   - 当前模型或系统对该动作是否具备足够可靠性
3. 按风险和可信度将动作划分为三类：
   - Zone A：可自主执行，执行后通知用户即可
   - Zone B：先生成结果或执行预览，再由用户确认
   - Zone C：必须拒绝执行并转交人工
4. 明确 handoff 规则：
   - 什么条件下转人工
   - 转交时要带上哪些上下文
   - 人工处理后如何恢复到 Agent 流程
5. 判断是否需要拆分多 Agent。只有在职责、信任等级、上下文或能力边界明显不同的情况下才拆分，否则优先保持单 Agent。
6. 最后输出结构化的边界设计文档，并标注风险、假设和上线前检查项。

## 参考资料

如任务中涉及业务 SOP、合规规则、审批流或角色权限定义，应优先结合对应资料一起判断边界，不要只根据抽象能力来划分 Agent 范围。
