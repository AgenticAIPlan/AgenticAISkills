# 单位调研 Pipeline Skill
本 Skill 不是普通报告生成器，而是一个调研流水线。它必须先产出关键事实的证据（来源链接），再写报告；必须先过链接审计和事实准入，再进入成稿，以尽量确保关键事实有可核查来源，有助于在交流前深入了解对方机构的情况和业务情况；必须通过 Reviewer 终审，才能标记为正式报告。

## 业务价值

本 Skill 的核心价值不只是生成机构背景报告，而是帮助各领域业务同学在与各类客户会议交流前，可以实现快速、可靠地理解一个机构的管理情况和业务情况。它面向会前沟通、合作研判、客户拜访、政府事务、产业研究和跨团队信息同步等场景，提供两类关键价值：

- **快速建立机构认知**：通过基本介绍、主要职责、隶属关系图示、内设机构、负责人、近期动态和业务相关部门等固定章节，帮助用户快速看清机构定位、权责边界、组织关系和沟通切入点。
- **显著降低模型幻觉风险**：通过候选来源、原子事实、链接审计、事实准入和 Reviewer 终审形成完整证据链，要求关键事实均可回到公开来源复核，避免无来源结论、虚假链接和模型记忆进入正式报告。

在此基础上，本 Skill 进一步解决机构调研中最容易反复出现的五类问题：
- 报告内容不可查证，无法判断是大模型幻觉还是真实信息。
- 虚假链接、首页链接、频道页链接进入正文，导致报告无法复核。
- 机构简介、职能职责、组织架构、负责人信息混入无来源判断。
- 把业务栏目、项目、活动新闻误写成现行内设机构或稳定隶属关系。
- 多单位或多轮调研时格式不一致，后续同事无法沿证据链继续补查。

目标交付不是“看起来完整”的长报告，而是可核查、可降级、可复用的会前机构情报：每个关键事实都能回到候选来源、原子事实、链接审计和事实准入表中复核，尽量确保关键事实真实、可核查，便于业务同学基于这些公开事实做出业务判断，以及基于有可能开展合作的方向，可以方便快捷地通过备注的来源链接进一步深入了解，以寻找业务合作机会。

业务收益：

- **提高调研效率**：将官网检索、来源筛选、事实抽取、链接检查和报告成稿固化为同一套流程，减少从零搭建调研框架的时间。
- **降低协作成本**：不同同事产出的报告使用相同字段、附录和准入标准，便于接力补查、复用和交叉复核。
- **支持业务判断**：把主管关系、组织架构、负责人、近期动态和业务相关部门放在同一证据链中，帮助判断沟通路径、审批链条和合作切入点。
- **提升风险控制**：通过事实准入和 Reviewer 终审，显式区分准入事实、待核实线索和剔除信息，避免无来源结论进入正式材料。

## 适用场景

当用户要求了解、拜访、会见、调研某个政府单位、企事业单位、研究机构、协会、基金会、科技馆、智库或其他机构时使用。

## 不适用场景 / 边界

本 Skill 保持独立完整，但不应被泛化为所有研究任务。以下情况不使用或必须降级：

- 只要求政策、行业或议题背景，不需要围绕具体单位建立事实链。
- 要求无来源内部判断、非公开信息、私人信息或未公开负责人分工。
- 要求直接生成合作建议、商业方案或沟通策略，但缺少单位事实证据；此时应先完成机构调研，再另行生成建议。
- 要求法律、投资、审计、合规结论等高风险判断；本 Skill 只提供公开事实背景和待核实线索。
- 只有主站首页、搜索结果、登录页、无法访问页面或无关页面时，不生成正式报告。

## 输入

- `单位名称`：必填；可一次输入多个单位。
- `会议主题`：可选；用于排序重点，不得用于创造无来源事实。
- `单位类型`：可选；政府机构 / 事业单位 / 企业 / 社会组织 / 自动判断。
- `交付形式`：可选；完整报告 / 简版正文 + 完整附录 / 仅输出证据表。默认完整报告。

默认执行完整高可靠调研流程。除非用户明确要求简版或只要证据表，否则不得省略候选来源、原子事实、链接审计、事实准入和 Reviewer 终审。

## Step 0：输入与默认假设

若用户未提供完整信息，不要阻塞执行，但必须在报告开头标注默认假设：

- 会议主题缺失：默认做通用会前背景调研。
- 单位类型缺失：先自动判断，并在候选来源表中说明判断依据。
- 交付形式缺失：默认完整报告。

只有当用户目标会影响检索范围且无法合理默认时，才提出简短澄清问题。

## 最小合格交付标准

标记为正式报告前，必须同时满足：

- 至少覆盖官网或官方介绍页、职责或业务说明、机构设置或组织关系、负责人信息、近期动态五类检索方向；确无公开来源时必须写入缺口说明。
- 每个进入正文的关键事实都能在原子事实证据表中找到对应行。
- 每个正文链接都通过链接审计，并在事实准入表中标记为 `准入`。
- 附录 A-E 齐全；如果用户要求简版正文，附录可另存为单独文件，但不得省略。
- Reviewer 终审不存在未修复 `error`。

## 必读文件

按需加载下列文件，不要一次性全文展开到最终报告中：

1. `references/source-rules.md`：来源纪律、官网优先级、推导边界。
2. `references/search-paths.md`：不同单位类型的检索路径。
3. `references/evidence-gate.md`：原子事实与事实准入规则。
4. `references/link-audit.md`：链接审计和页面相关性校验。
5. `references/org-structure.md`：隶属关系、内设机构、关联单位梳理规则。
6. `references/leader-research.md`：主要负责人调研规则。
7. `references/recent-news.md`：近期动态筛选规则。
8. `references/final-review-checklist.md`：Reviewer 终审清单。
9. `references/example-good-report.md`：高质量输出示例导读。
10. `references/example-nda-good-report.md`：基于国家数据局真实评测案例整理的正式报告样例。
11. `references/common-failure-cases.md`：常见失败案例和修正方式。

输出模板位于：

- `assets/candidate-sources-template.md`
- `assets/atomic-evidence-template.md`
- `assets/link-audit-template.md`
- `assets/fact-admission-template.md`
- `assets/report-template.md`
- `assets/reviewer-output-template.md`

辅助脚本位于：

- `scripts/link_audit.py`：从候选来源表、证据表或报告草稿中提取 Markdown 链接并生成链接审计表。脚本只做可访问性和基础页面信息检查，页面是否支持事实仍需按 `references/link-audit.md` 人工或模型复核。HTTP 200 / 可访问只代表链接可打开，不代表页面支持对应事实，也不代表事实准入。

如果任一必需文件无法读取，必须在输出中说明缺失文件，并降级为“阶段性调研稿”。

## Pipeline

每个单位必须独立跑完整流水线。多单位调研时，总报告状态取最低状态：只要任一单位为阶段性调研稿，总报告即为阶段性调研稿。

### Step 1：候选来源收集

目标：只收集候选来源，不写报告结论。

必须输出 `候选来源清单`，字段使用 `assets/candidate-sources-template.md`。

覆盖范围：

- 官网或官方介绍页。
- 职能职责、机构设置、领导信息、新闻动态。
- 上级主管、股东、实际控制人、直属单位、主管社会组织、关联组织。
- 业务指导、技术归口、标准化技术委员会、专家委员会、咨询委员会、秘书处承办、联盟 / 联合实验室 / 工作组等非行政隶属关联。
- 招聘、招标、预算、年报、事业单位登记、工商平台、社会组织平台。
- 近 1 年核心业务动态。

门禁：没有候选来源清单，不得进入 Step 2。

### Step 2：原子事实提取

目标：从候选来源中提取原子事实，不做段落写作。

必须输出 `原子事实证据表`，字段使用 `assets/atomic-evidence-template.md`。

规则：

- 一个事实一行。
- 原子事实证据表必须按 `来源编号` 排序；同一来源下再按信息点或事实编号排序，便于逐源核查。
- 不把多个判断压成一行。
- 不把“曾出现”写成“现设有”。
- 不把会议主题、行业常识或模型记忆变成事实。

门禁：正文将来只能使用此表中的事实；如果写作时发现事实不足，必须回到 Step 1 或 Step 2 补证。

### Step 3：链接审计

目标：逐一访问所有 URL，检查可访问性和页面相关性。

若当前环境可以运行本地脚本，优先使用 `scripts/link_audit.py` 辅助生成初版链接审计表。例如：

```bash
python skills/org-research/scripts/link_audit.py --input draft.md --output link-audit.md
```

必须输出：

- `链接审计表`，字段使用 `assets/link-audit-template.md`。
- `剔除 / 降级链接清单`。

通过条件：

- 链接状态为 `200`，或在受限网站场景下记录明确状态枚举并说明可验证方式。
- 页面标题或正文支持对应原子事实。
- 非首页、非频道页、非搜索结果页、非登录页、非无关页面。

注意：HTTP 200 / 可访问只是最低技术条件，不等于事实准入；页面必须能支持对应原子事实，且仍需进入 Step 4 判断是否准入。

门禁：404、NotFound、Timeout、DNS Failed、Redirect to Homepage、Login Required、内容不相关的来源不得进入正式正文。

### Step 4：事实准入

目标：决定哪些原子事实可以进入正文。

必须输出 `事实准入表`，字段使用 `assets/fact-admission-template.md`。

准入结果只能是：

- `准入`：可进入正文。
- `待核实`：只能进入待核实线索。
- `剔除`：不得进入报告。

门禁：报告撰写阶段禁止新增事实；只能消费 `准入` 事实。

### Step 5：组织分析

目标：基于准入事实生成组织关系草图和缺口清单。

必须输出：

- 向上隶属关系。
- 向下关联关系：内设机构、直属单位、控股参股企业、主管社会组织、联合机构、标委会。
- 非行政隶属关联：业务指导单位、技术归口单位、专家咨询委员会、标准化技术委员会、秘书处承担单位、联盟、联合实验室、工作组等；必须标明关系类型，不得写成直属下属。
- 完整性判断：完整 / 部分确认 / 无法确认。
- 会议主题相关部门或业务方向，但不得新增无来源部门。

门禁：无法确认完整性时，必须明示缺口，不得写成完整清单。向上溯源默认遵循“业务必要披露”原则：若更高层级主管机关或出资 / 管理层级对会前沟通、合作判断、审批链条、资源协调或单位性质判断必要，可以进入组织分析和最终图示，并说明业务相关性；若仅属于泛背景信息且与当前任务无直接关系，默认保留在证据层，不进入正文图示。

### Step 6：报告生成

目标：使用 `assets/report-template.md` 生成正式报告或阶段性调研稿。

硬规则：

- 只能使用 Step 4 中 `准入` 的事实。
- 待核实事实只能进入“待核实线索”。
- 正文中的 URL 必须使用 Markdown 链接格式 `[来源标题](URL)`，不得裸露长 URL。
- 附录表格中的 URL 也必须使用 Markdown 链接格式 `[来源标题](URL)`；不得在表格单元格中裸露长 URL。
- 来源标注必须放在段落或条目末尾。
- 来源类型标注只能单选一个：`「官网原文」`、`「总结提炼」`、`「侧面推导」`、`「新闻拼图」`、`「官网 API 原文」`。不得写成 `「官网原文 / 总结提炼」` 这类组合标签。
- `主要职责` 列表中，每条职责名称必须加粗，例如 `- **科技活动与学术交流服务**：...`。
- `隶属关系图示` 默认只展示完成当前任务所必需的组织关系；若展示更高层级主管机关或出资 / 管理层级，必须说明其对会前沟通、合作判断、审批链条、资源协调或单位性质判断的业务必要性。
- `隶属关系图示` 中已确认的内设机构、下属单位、投资企业、标委会、学会协会等清单不得使用“等”省略；若无法列全，必须标注为“部分确认”并说明缺口。
- `内设机构` 章节必须按机构类型使用无序列表展示，不得把多个机构连续写在一个段落中。
- `主要负责人` 章节必须使用无序列表展示，每位负责人单独一条；若缺少公开履历或分工，在列表后单列缺口说明。
- 多单位报告必须逐单位标注报告状态。
- 若完整证据表不展示，报告不得标为正式报告。

### Step 7：Reviewer 终审

目标：使用 `references/final-review-checklist.md` 逐条审查输出。

必须输出 `Reviewer 终审结果`，字段使用 `assets/reviewer-output-template.md`。

问题等级：

- `error`：违反硬规则，必须修复；若未修复，报告状态必须为阶段性调研稿。
- `warning`：不影响基本可用，但必须提醒用户。
- `info`：优化建议。

门禁：只要存在未修复 `error`，不得输出正式报告。

## 最终输出顺序

最终答复或生成文件必须按以下顺序组织：

1. 调研报告正文。
2. 待核实线索。
3. 信息来源汇总。
4. 附录 A：原子事实证据表。
5. 附录 B：候选来源清单。
6. 附录 C：链接审计表。
7. 附录 D：事实准入表。
8. 附录 E：Reviewer 终审结果。

如用户要求简版报告，可压缩正文篇幅，并将附录另存为单独文件，但必须在正文中给出文件路径；不得省略候选来源、原子事实、链接审计、事实准入和 Reviewer 终审。简版只能简化展示形式，不能降低事实门禁。

## 禁止事项

- 禁止用主站首页、频道页、搜索结果页冒充具体事实来源。
- 禁止用不可访问 URL 或 NotFound 页面作为来源。
- 禁止自检写“部分是”并算通过；`部分` 等同于 `否`，必须触发降级或 Reviewer warning/error。
- 禁止为了凑够近期动态数量而加入无具体日期、无具体 URL 或与核心业务无关的动态。
- 禁止把新闻拼图写成官网领导班子。
- 禁止把业务板块、展厅、项目、栏目写成行政内设机构。
- 禁止在正文裸露长 URL；必须使用 `[来源标题](URL)`。
- 禁止在附录表格中裸露长 URL；URL 字段也必须写成 `[来源标题](URL)`。
- 禁止将互斥来源标签合并使用，例如 `「官网原文 / 总结提炼」`。
- 禁止在已确认完整清单的 `隶属关系图示` 中用“等”代替列全。
- 禁止将 `内设机构` 或 `主要负责人` 写成大段逗号分隔文本。
